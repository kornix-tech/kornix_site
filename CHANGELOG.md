# Changelog

## [Unreleased]

### Fixed
- Frontend runtime жёстко переведён на backend session/API: удалены
  синтетические auth/data builders и env-переключатели автономного режима,
  а сбой проверки сессии теперь показывает "отсутствует связь с сервером".
- BFF auth/session/CSRF frontend client switched to canonical `/api/v2/*`,
  fixing the `API 404: Not Found`
  authorization check in local front+back-via-API mode.
- Frontend runtime очищен от удалённых переключателей и внутреннего имени
  группы расчёта: пользовательский calculation API теперь canonical, а
  soil-water метрики сгруппированы как `soil_water`.
- API client стал устойчивее к неполному backend DTO: пустые `features`,
  `metrics`, `recommendations`, `methods` и `catalog.fields` больше не приводят
  к падению страницы при нормализации ответа.
- CSRF bootstrap и retry после `CSRF_TOKEN_INVALID` теперь используют тот же
  timeout-механизм, что и остальные frontend API-запросы.

### Technical
- Production nginx proxy получил таймауты для длинных approval/recalculation
  API-запросов, согласованные с frontend calculation timeout.

### Added
- Добавлена frontend-поддержка 44 метрик FAO90 single-layer soil chain для
  `simple_eto_single_layer_soil`: динамический CSV/export всех profile metrics,
  grouped FAO90 summary/diagnostics в графике, расширенный map tooltip/export
  и live smoke `scripts/frontend_eto_single_layer_soil_fao90_metrics_smoke.mjs`.
- Frontend editable approval UAT smoke обновлён под backend handoff
  `63c699da5e2c30a6d31f6011384e4d748ab7dbdb`: READY теперь доказывает
  production frontend origin `http://localhost:8080`, same-origin `/api/*`,
  session auth, editable context, approval POST/readback и cleanup ephemeral
  пользователя без записи секретов.
- Добавлен финальный frontend editable approval UAT smoke для live BFF/proxy
  режима: проверяются ephemeral auth, `current_editable`, 37 map features,
  13 profile metrics с shortwave, approval POST/readback и негативные CSRF/scope
  сценарии без записи секретов в отчёты.
- Добавлен pre-UAT same-origin browser/proxy smoke для проверки, что frontend
  origin обслуживает SPA и проксирует `/api/*` в backend JSON/API, а не в
  `index.html`.
- Добавлен ephemeral auth mode для frontend API v2 SP37 live-smoke: при
  отсутствии внешних smoke credentials runner создаёт временного backend
  пользователя через существующий bootstrap helper, проходит обычный
  CSRF/login/session flow и затем отзывает сессии и деактивирует пользователя
  без записи пароля в отчёты.
- Добавлен frontend API v2 SP37 live-smoke runner для проверки опубликованного
  `currentAppliedCalculationRunId`, 37 map features и 13 profile metrics без
  вывода cookies, CSRF token или пароля в отчёты.
- Добавлена dependency-free проверка покрытия profile metrics, блокирующая
  потерю backend metric `shortwave_radiation_daily_mj_m2` в графике или CSV.
- Добавлен свежий baseline-аудит frontend-кодовой базы KORNIX с машинно-читаемыми
  отчётами в `codex_reports/` и ссылками на текущий security/documentation
  статус в README.

### Changed
- Рабочие разделы frontend получили человекочитаемые URL с организацией и
  сезоном: `/fields/sp/2026`, `/water-regime/sp/2026` и
  `/irrigation-input/sp/2026`; технический `calculationRunId` больше не
  выводится в канонический URL.
- Атмосферный блок графика водного режима получил разнесённые по высоте шкалы:
  линии температуры, влажности, ветра, испарения и радиации визуально сжаты и
  больше не растягиваются каждая на всю высоту блока.
- Tooltip всех блоков графика водного режима теперь сортирует строки по
  фактическому положению линий и областей сверху вниз в выбранный день.
- Все блоки графика водного режима используют единый tooltip с непрозрачной
  подложкой, чтобы подписи не смешивались с линиями графиков.
- Y-шкалы графиков водного режима больше не строят линии сетки и подписи по
  дробным значениям.
- Диапазон управления влагозапасами теперь считается как 60—90% доступной влаги
  между НВ и TAM, TAM выводится отдельной чёрной линией, а текущие влагозапасы
  рисуются ломаной линией без сглаживания.
- График влагозапасов переведён в координаты дефицита от НВ: НВ отображается
  как `0` на верхней границе, а TAM, диапазон управления и текущая влажность
  откладываются вниз в миллиметрах от НВ.
- В правую панель графика добавлены пользовательские настройки диапазона
  регулирования в долях НВ с шагом `0.01`, значениями по умолчанию `0.60—0.90`
  и проверкой минимального разрыва `0.05`.
- Настройки диапазона регулирования влагозапасов сохраняются как frontend-
  предпочтение, привязанное к организации и учётной записи пользователя.
- Правая панель графика переупорядочена: настройки диапазона, календарный
  диапазон, прогноз, легенда, модель, метрики модели и экспорт теперь идут в
  фиксированной рабочей последовательности.
- В правой панели добавлен заголовок `Отображаемые даты`, а поля диапазона
  регулирования оформлены в стиле календарных полей.
- Типографика правой панели графика унифицирована, а легенда приведена к
  фактическим линиям и областям графиков, включая НВ, ВЗ и диапазон
  регулирования.
- Легенда графиков получила единицы измерения и интерактивную подсветку:
  при наведении графики затемняются, а выбранный по легенде ряд остаётся
  выделенным своим цветом.
- Скорость ветра в атмосферном блоке теперь отображается тонкой тёмно-зелёной
  линией, а влажность воздуха — синей штриховой линией; легенда синхронизирована
  с этими стилями.
- В блоке растений суммарное испарение отображается толстой зелёной линией, а
  испарение почвы и растений — тонкими штриховыми коричневой и травянисто-
  зелёной линиями.
- Все четыре графических блока комплексного графика теперь имеют одинаковую
  высоту во всех рабочих раскладках.
- Легенда глубины поливов на вкладке ввода поливов теперь по умолчанию
  выключена при входе, а включённое пользователем состояние сохраняется только
  до logout.
- Левая sticky-панель вкладки ввода поливов приведена к оформлению списка полей
  вкладки `Водный режим`: добавлен поиск поля с фильтрацией по номеру, названию
  культуры и сорту, а карточки используют единые подписи без служебного префикса
  организации.
- Горизонтальная линейка прокрутки таблицы ввода поливов визуально ограничена
  календарной частью и больше не проходит под левой плашкой списка полей.
- Карточки полей в списке водного режима теперь окрашиваются по тому же
  forecast-статусу `today+7`, что и индикатор справа, чтобы фон карточки
  соответствовал зоне графика влагозапасов для поля.
- Индикатор справа на карточке поля теперь окрашивается по frontend-зоне
  графика влагозапасов на последний день прогноза: жёлтая зона вне диапазона
  регулирования, зелёный диапазон регулирования, красная зона ниже ВЗ.
- Фон карточки поля в списке водного режима теперь окрашивается по frontend-зоне
  графика влагозапасов на текущую отображаемую дату, отдельно от forecast-
  индикатора справа.
- Левая плашка списка полей в водном режиме расширена и больше не показывает
  горизонтальную прокрутку: карточки сжимаются по доступной ширине.
- На вкладке `Ввод поливов` sticky-колонка полей заменена отдельной левой
  панелью в стиле `Водного режима`; календарь ввода поливов расположен справа
  и прокручивается как отдельная календарная область.
- На вкладке `Ввод поливов` левая плашка полей поднята к верхней границе
  рабочей области, а легенда глубины поливов сдвинута вправо над календарём.
- На вкладке `Ввод поливов` календарь опущен под служебную полосу легенды и
  кнопки утверждения, шапка календаря сделана выше, включена горизонтальная
  прокрутка календарной области, а строки календаря выровнены по карточкам
  полей.
- Карточки полей в левых плашках стали ниже по вертикали, а зазор между ними
  уменьшен для более плотного списка.
- Hover-подсказки цветных карточек полей приведены к цвету зоны влагозапасов:
  розовая — `Нужен полив`, жёлтая — `Требуется анализ`, зелёная —
  `Влагозапасы в норме`.
- Левая плашка `Поля` вынесена в единый frontend-компонент списка полей и
  используется на вкладках `Водный режим` и `Ввод поливов` с общей логикой
  поиска, выбора, окраски по зоне влагозапасов, hover-подсказок и forecast-
  индикатора.
- У вкладки `Ввод поливов` удалена отдельная CSS-геометрия левой плашки:
  список полей теперь использует те же размеры, padding и шаг карточек, что и
  плашка на вкладке `Водный режим`.
- Вкладка `Ввод поливов` приведена к компоновке `Водного режима`: левая плашка
  полей и календарный элемент теперь стоят рядом как отдельные элементы на
  рабочем холсте, без общей расширенной рамки календаря вокруг списка полей.
- Календарь ввода поливов оформлен как отдельный белый прямоугольник: в верхней
  части карточки расположены справочная информация и кнопка `Утверждаю`, ниже
  идёт шапка календаря, затем синхронизированные с карточками полей строки.
- В календарной карточке ввода поливов добавлены узкие внутренние отступы по
  левому и правому краям, чтобы сетка календаря не прилипала к рамке.
- На вкладке `Ввод поливов` шаг календарных строк привязан к высоте карточек
  полей: границы строк календаря проходят по центру зазора между карточками.
- В нижнюю часть календаря ввода поливов добавлена отдельная горизонтальная
  линейка прокрутки, синхронизированная с прокруткой таблицы.
- В комплексном графике скрыты подписи вертикальных осей справа и слева от
  блоков, чтобы шкалы не перегружали рабочую область.
- После скрытия вертикальных шкал графические зоны расширены почти до краёв
  блоков с небольшим внутренним отступом.
- Линии и области прогноза в комплексном графике теперь начинаются от последней
  расчётной точки перед прогнозом, чтобы на границе `today/forecast` не было
  визуального разрыва.
- График влагозапасов получил фоновые зоны: жёлтые области выше и ниже
  диапазона регулирования, зелёный диапазон регулирования, красную область
  ниже ВЗ, тёмно-коричневую линию НВ и оранжевую линию ВЗ.
- Во всех блоках графика добавлен узкий внутренний отступ от краёв, а красная
  область влагозапасов ниже ВЗ доведена до нижней дизайнерской границы.
- Прогнозная часть графика влагозапасов теперь визуально отделяется как в
  остальных блоках: заливки стали бледнее, а линии ВЗ и влагозапасов —
  штриховыми и приглушёнными.
- Tooltip графика влагозапасов заменён на предметные подписи: НВ, диапазон
  регулирования в долях НВ, текущие влагозапасы и доступные влагозапасы.
- В зоне растений графика водного режима теперь отдельно выводятся фактическая
  эвапотранспирация `actual_evapotranspiration_mm` и транспирация культуры
  `actual_transpiration_mm`; ET собирается из транспирации и испарения почвы
  приоритетно, а готовая `actual_evapotranspiration_mm` используется только
  если обе составляющие отсутствуют. Legacy `crop_transpiration_daily_mm`
  больше не используется как fallback для транспирации культуры.
- В зоне растений добавлена отдельная линия `actual_soil_evaporation_mm`;
  если backend не передал транспирацию, но передал испарение почвы, транспирация
  отображается как `0`, а суммарная ET остаётся видимой.
- Legacy `crop_transpiration_daily_mm` используется только как fallback для
  суммарной ET/испарения почвы, чтобы API-стенды без ненулевых `actual_*`
  не оставляли зону растений пустой.
- В списке полей скрывается служебный префикс организации в номере поля
  (`SP.1.1` → `1.1`), а поиск и подпись культуры учитывают сортовые поля
  backend-ответа, если они переданы отдельно от `cropName`.
- FAO90 diagnostics в боковой панели графика теперь выбирают ближайший день
  профиля с реальными значениями, чтобы не показывать `нет данных` при
  доступных backend-метриках.
- Служебные backend-warning коды готовности и качества данных больше не
  выводятся в пользовательских панелях карты, метода и графика.
- Диапазон управления на графике влагозапасов теперь строится от `0.6 НВ` до
  `0.9 НВ`, при этом линии НВ и ВЗ берутся из отдельных backend-метрик.
- FAO90 frontend metrics report gate приведён к согласованному состоянию:
  устаревший pre-commit git-блок больше не описывается как финальная delivery
  metadata, добавлены отдельные `report_gate` JSON/MD отчёты.
- Таблица ввода поливов теперь блокирует редактирование при любом состоянии,
  кроме `frontendMode=current_editable` и `submitAllowed=true`, чтобы UI не
  оставлял активные поля при backend-issued запрете submit.
- Frontend API client теперь явно передаёт `seasonYear` в поддерживающие API v2
  endpoints `current-context`, `readiness/current`, `irrigation-layer/current`
  и `field-seasons/catalog`, чтобы URL/state сезона не зависел от backend
  default.
- Удалены устаревшие справочные документы, не соответствующие текущему
  UAT/production contract для расчётных данных `/api/v2/kornix/*`.
- График водного режима теперь потребляет и экспортирует солнечную радиацию
  `shortwave_radiation_daily_mj_m2` (`МДж/м²/сутки`) как отдельную линию в
  weather-зоне, чтобы backend SP37 profile response из 13 метрик не сжимался до
  12 видимых/exported метрик.
- Frontend calculation-run status helper переведён на production route
  `/api/v2/kornix/water-regime/calculation-runs/{calculationRunId}`; старый
  `/api/v2/kornix/calculation-runs/{calculationRunId}/status` удалён из
  runtime-кода и README-контракта.
- Frontend подготовлен к VDS production security: production API base теперь
  документирован и проверяется как `/api`, добавлены `Dockerfile.prod`,
  `docker-compose.prod.yml`, `.env.local.example` и `.env.production.example`.
- BFF login UX переведён с redirect-сценария на форму `username/password` с
  `POST /api/v2/auth/login`, последующим refetch `/api/v2/me` и сохранением
  запрета на хранение токенов в browser storage.
- Production Nginx CSP ужесточён: удалены localhost API origins из `connect-src`,
  оставлены security headers и задокументирована причина временного
  `style-src 'unsafe-inline'`.
- CSRF handling для unsafe API-запросов теперь выполняет один безопасный
  refresh/retry при backend code `CSRF_TOKEN_INVALID`.
- Добавлена обязательная frontend security documentation в `doc/security/` и
  расширен contract-check для VDS security readiness.
- Локальный integration-dev профиль сохраняет `VITE_API_BASE_URL=http://localhost:8001`,
  но в Docker/Vite dev-режиме проксирует same-origin `/api/*` к backend через
  `host.docker.internal:8001`, чтобы frontend smoke не зависел от CORS.
- Approval submit для API v2 больше не отправляет backend-only metadata
  `managedScope.scopeHash`; payload нормализуется до строгих полей
  `dateFrom`, `dateTo`, `fieldSeasonIds`, `scopeVersion`.
- Ввод поливов для API v2 теперь загружает backend active projection через
  `/api/v2/kornix/irrigation-layer/current`, использует его как начальное
  состояние таблицы и считает `clientDiff` от backend projection, а не от
  локальной отметки утверждения.
- Editable range таблицы поливов ограничен backend-issued `managedScope`
  (`dateFrom..dateTo` и список `fieldSeasonIds`); значения вне scope
  блокируются и не могут попасть в approval submit.
- Contract-check дополнительно запрещает `/api/admin/v1`, `/admin` leakage и
  локальную `approvedSignature`-семантику для утверждённых поливов.
- Пользовательский workspace KORNIX использует контракт `/api/v2/kornix`;
  auth/me и CSRF также используют canonical `/api/v2`.
- Отображаемый расчёт теперь определяется `currentAppliedCalculationRunId` из
  backend current-context; старое предположение про latest completed alias
  удалено из рабочего flow.
- Добавлен выбор метода расчёта из `availableMethods`; map/profile запрашивают
  данные с выбранным `methodCode`, а URL хранит только валидный выбранный метод.
- Утверждение поливов переведено на approval workflow: frontend отправляет
  `managedScope`, положительный `irrigationLayer`, `clientDiff` и обрабатывает
  polling статуса approval без переключения на failed расчёт.
- `stale_read_only` и `not_ready` режимы current-context блокируют отправку и
  редактирование поливов, но не трактуются как ошибка авторизации.
- Интеграционный env-профиль описывает canonical `/api/v2` auth endpoints и
  v2-only `/api/v2/kornix` расчётов.
- В validate-проверку добавлен contract-check, который блокирует возврат
  retired KORNIX routes, старого calculate-flow и `irrigation_tasks` в рабочем
  `src`.
- Старый синтетический `calculationRunId` на рабочей странице удалён из
  runtime-пути вместе с автономным frontend data fallback.
- BFF auth больше не маскирует сетевые ошибки `/api/v2/me` под гостевую
  сессию: реальные ошибки backend показываются как сбой проверки авторизации.
- Календарная арифметика карты, графика и ввода поливов переведена на
  timezone-neutral ISO-дни, чтобы не зависеть от локальной зоны браузера.
- Шаговые стрелки ввода полива больше не создают значение `0`: нижняя граница
  после пустого значения — `1` мм.
- Черновики и признак утверждения поливов в `localStorage` теперь разделены по
